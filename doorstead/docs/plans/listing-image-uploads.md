# Plan: Listing image uploads

Source: docs/prds/listing-image-uploads.md
Feature key: listing-image-uploads  (one feature = this plan; issues get label `feat:listing-image-uploads`, work lands on branch `feat/listing-image-uploads`)

## Module interaction

Direction is one-way, matching `doorstead/CLAUDE.md` (route to service contract to db, never reversed). An admin route or server action imports the `MediaService` contract (upload, list, gallery resolution, reorder, retag, remove) and the unchanged `ListingService` contract; it never imports `@supabase/*`. The `MediaService` implementation calls the `StorageAdapter` contract (logic-free put/remove bytes plus a signed read URL) and a media repository through the cookie-aware server client; the `StorageAdapter` implementation is the only caller of the Supabase Storage SDK, exactly as `lib/db/*` is the only caller of the DB driver. Image transformation (web-optimised copy plus thumbnail) lives behind the `MediaService` via an `ImageTransformer` seam, so the caller never learns a variant was made. The publish predicate (`lib/listings/schema.ts` plus its two callsites in `lib/listings/actions.ts`) is widened to count uploaded media via the `MediaService` contract, never reading the table directly. Image resolution for every page lives behind one `MediaService` boundary that internally falls back to a listing's legacy `photoUrls`, so no page ever branches on `ListingMedia[] | string[]`. `Listing.photoUrls` stays on the contract through the whole feature. UI components in `components/**` import no service and no DB; they receive resolved `GalleryImage[]` as props and render through one path.

## Tracer bullet

One admin uploads one image file through a real `uploadListingImage` server action; the action gates on `requireAdmin()`, the service validates the bytes by content type and magic-byte sniff, the adapter writes the original to a private Supabase Storage bucket under a server-generated opaque key, one `listing_media` row is written last (insert-last, with a best-effort compensating remove if the insert fails), and the public listing page renders that stored image from a signed URL. Thinnest path through every layer: file input to server action to `MediaService` to `StorageAdapter` to db to public gallery `<img>`. No variants, no drag-order, no cover or floorplan tag yet; those are later units, each adding one dimension of difficulty. This is Unit 1. It is the unit that first produces a real `listing_media.id` consumable by every later save-side path, which resolves the ordering tension where cover/floorplan persistence would otherwise consume ids no real path produces.

## Unit 1: Tracer, upload one file end to end through a real action

- What it builds: an admin attaches a single image file and submits it to a real `uploadListingImage` server action; the action enforces `requireAdmin()`, the service validates type by content type plus a magic-byte sniff, the adapter stores the original in a private bucket under an opaque server key, one media row is written last, and the image renders on the public listing page through an `<img>` fed a signed URL, touching every layer with the least new surface. This unit is what first mints a real `listing_media.id`.
- Modules + contracts:
  - `lib/media/contract.ts`: `interface UploadFile { bytes: ArrayBuffer; filename: string; contentType: string; sizeBytes: number }`; `interface ListingMedia { id: string; listingId: string; storageKey: string; originalUrl: string; position: number }`; `type UploadResult = { ok: true; media: ListingMedia } | { ok: false; reason: 'invalid_type' }`; `interface MediaService { uploadOriginal(listingId: string, file: UploadFile): Promise<UploadResult>; listForListing(listingId: string): Promise<ListingMedia[]> }`. `originalUrl` is a signed read URL minted at read time, not a stored column.
  - `lib/media/storage/contract.ts`: `interface StorageAdapter { put(key: string, bytes: ArrayBuffer, contentType: string): Promise<void>; signedUrl(key: string): Promise<string>; remove(key: string): Promise<void> }`. `signedUrl` uses the storage default TTL (no custom expiry, staying within PRD out-of-scope). `put` returns nothing; the key is server-supplied, so no URL flows back from the write.
  - `lib/media/actions.ts`: `async function uploadListingImage(formData: FormData): Promise<UploadResult>` that calls `authService.requireAdmin()` as its first line, reads the file, and delegates to `MediaService.uploadOriginal`. An unauthenticated call rejects before any storage or db touch.
  - Key derivation (stated, not a public signature): the service builds the storage key as `${listingId}/${randomUUID()}.${ext}` where `ext` derives from the validated content type (`png`|`jpg`), never from the attacker-controlled `filename`. `filename` is metadata only.
  - Atomicity (stated contract): `uploadOriginal` is insert-last. It calls `StorageAdapter.put` first, then writes the `listing_media` row; if the row insert throws, it calls a best-effort `StorageAdapter.remove(key)` to compensate, then surfaces the failure. No media row is ever written without its bytes, and a failed insert leaves no referenced key. Residual-orphan risk (a `remove` that itself fails) is acknowledged; a background sweep is out of scope for this feature.
  - `supabase/migrations/0003_listing_media.sql`: new `listing_media` table (`id` uuid pk, `listing_id` uuid fk references `listings(id)` on delete cascade, `storage_key` text, `position` int, `created_at`). RLS on the table mirrors `listings` (migration 0001 idiom): anon SELECT only where the parent listing is live and not deleted; authenticated ALL only when `auth.uid() in (select user_id from admins)`. A private storage bucket `listing-media` (not public), plus `storage.objects` policies, since under the anon-key runtime every write passes RLS as the authenticated admin and Storage objects are governed on a separate `storage.objects` surface the table policies do not cover: INSERT, UPDATE, DELETE policies on `storage.objects` gated on `bucket_id = 'listing-media' and auth.uid() in (select user_id from admins)`; SELECT on `storage.objects` left to authenticated admins only (no anon object SELECT) because reads are served through short-lived signed URLs, keeping draft-listing media private and consistent with how `listings` hides drafts.
- Seams: one new seam, the `StorageAdapter` contract. Tests drive `MediaService` through a `FakeStorageAdapter` (in-memory map of key to `{ bytes, contentType }`, plus a deterministic `signedUrl`), asserting observable fake state (a key is present in the map, a row is returned carrying that key, `originalUrl` is the fake signed URL), never asserting the SDK was called. The action is tested through a `FakeMediaService`, asserting the `requireAdmin`-throws path returns the typed reject and never reaches the service. The thin `supabase-storage.ts` adapter has no logic to test. The write path runs under the anon key, RLS-gated by the `storage.objects` policies; no service-role key in the runtime.
- Acceptance: uploading one valid PNG or JPEG through `uploadListingImage` (backed by `FakeMediaService` / `FakeStorageAdapter`) returns `{ ok: true }` with a `ListingMedia` whose `storageKey` matches the `${listingId}/<uuid>.<ext>` shape and whose `originalUrl` is the fake signed URL, the key is present in `FakeStorageAdapter`'s map, and `position: 0`; `listForListing` returns it; an `uploadOriginal` test with `FakeStorageAdapter` configured to throw on the row insert leaves no key in the map (compensating `remove` ran) and writes no row; an unauthenticated action call returns the typed reject (or throws `Unauthorized`) with no key written; the public listing page renders the signed `originalUrl` in an `<img>`.
- Blocked by: none, parallel-safe

## Unit 2: Server-side validation (type by magic bytes, size, count)

- What it builds: full server-side rejection of disallowed type, oversize file, and over-cap additions, each with a typed reason, so validation holds even when the client is bypassed. Type is gated by a magic-byte sniff, not by the spoofable content-type header.
- Modules + contracts:
  - extend `lib/media/contract.ts`: `type RejectReason = 'invalid_type' | 'too_large' | 'too_many'`; `interface MediaLimits { maxBytes: number; maxPerListing: number; allowedTypes: readonly string[] }`; `UploadResult` reason widened to `RejectReason`.
  - `lib/media/schema.ts` (pure functions, mirrors `lib/listings/schema.ts`): `function validateUpload(file: UploadFile, existingCount: number, limits: MediaLimits): { ok: true } | { ok: false; reason: RejectReason }`; `const DEFAULT_MEDIA_LIMITS: MediaLimits` with `maxBytes: 10 * 1024 * 1024`, `maxPerListing: 20`, `allowedTypes: ['image/png', 'image/jpeg']`. The type check sniffs the leading bytes (PNG `89 50 4E 47`, JPEG `FF D8 FF`) against the first bytes of `file.bytes`; `file.contentType` is treated as a hint, not the gate, so a PDF, HTML, or SVG renamed and labelled `image/png` is rejected as `invalid_type`.
- Seams: existing, pure-function tests on `validateUpload` (the house pattern already used for `schema.test.ts` and `photo-urls.test.ts`); plus `MediaService` through `FakeStorageAdapter` to prove a rejected file leaves no key in the fake map and no row, while valid siblings in the same batch still store.
- Acceptance: `validateUpload` returns `invalid_type` for a PDF, for WebP and HEIC, and for a non-image byte stream mislabelled `image/png` (magic bytes do not match), `too_large` for a file over 10 MB, `too_many` for the 21st image at cap 20, and `ok` for a valid in-bounds PNG or JPEG whose magic bytes match; a `MediaService` test confirms a rejected upload leaves `FakeStorageAdapter`'s map and the row set untouched while a valid sibling in the same batch is present in the map.
- Blocked by: Unit 1

## Unit 3: Variants, web-optimised copy and thumbnail, keep original

- What it builds: every successful upload also produces a web-optimised copy and a thumbnail, stored alongside the retained original; `ListingMedia` gains `webUrl` and `thumbUrl`. The transformer is pinned and its Vercel runtime envelope is stated.
- Modules + contracts:
  - extend `lib/media/contract.ts`: `ListingMedia` gains `webUrl: string; thumbUrl: string` (signed read URLs minted at read time, like `originalUrl`).
  - `lib/media/transform/contract.ts`: `interface ImageTransformer { toWeb(bytes: ArrayBuffer, contentType: string): Promise<{ bytes: ArrayBuffer; contentType: string }>; toThumb(bytes: ArrayBuffer, contentType: string): Promise<{ bytes: ArrayBuffer; contentType: string }> }`. The transformer returns the same content type it is given; resizes are same-format, no conversion.
  - Real transformer: `sharp` (pinned in `package.json`), run inline in the upload server action's Node runtime. Runtime envelope stated: the `uploadListingImage` route runs on the Node.js runtime with `maxDuration` raised to 60s and function memory at 1024 MB; expected latency for a single 10 MB PNG (the configured `maxBytes`) into web plus thumb is sub-second to low-seconds on a warm function, with sharp cold-start adding roughly 1 to 2s on the first call. The action processes one file per request (the client uploads files individually and shows per-file progress), so a 20-file batch is 20 independent requests, never one 20-file transform inside a single function invocation; this keeps each invocation inside the memory and duration budget and removes the OOM-on-batch risk. If a single 10 MB transform is later found to exceed this budget in practice, the variant step moves off the request path (store original synchronously, generate web/thumb in a queued job), which changes this unit's seams; that fallback is named but not taken pre-emptively.
  - `MediaService.uploadOriginal` widens internally to produce three keys (original, web, thumb) via the adapter; the contract verb is unchanged, depth grows behind it. Insert-last and the compensating `remove` from Unit 1 still hold and now compensate all keys put so far on any failure.
- Seams: existing `StorageAdapter` seam (now three `put` calls) plus the new `ImageTransformer` boundary. Drive `MediaService` through `FakeStorageAdapter` plus a `FakeImageTransformer` (deterministic byte stubs). The real `sharp` transformer is exercised only at its own boundary, never faked into the service. A partial-failure test configures `FakeStorageAdapter` to throw on the Nth `put` and asserts no `listing_media` row exists and none of the earlier keys remain in the fake map.
- Acceptance: a `MediaService` upload test (fake transformer plus fake storage) yields a `ListingMedia` with three distinct, populated signed URLs and three keys present in `FakeStorageAdapter`'s map, all the same format as the input, with the original key still present after variants are made; a test failing the second of three puts leaves zero rows and zero keys in the fake map.
- Blocked by: Unit 1

## Unit 4: Persist ordering, cover, and floorplan on save

- What it builds: the form's media set, its order, the cover, and the floorplan tag persist on listing save and reload on edit. Cover is exactly one (defaulting to first if unset); floorplan is at most one and independent of cover. The media ids consumed here are real ids minted by Unit 1's `uploadListingImage`, so the slice is exercisable end to end the moment it lands.
- Modules + contracts:
  - extend `lib/media/contract.ts`: `ListingMedia` gains `isCover: boolean; isFloorplan: boolean`; `interface MediaService` gains `reorder(listingId: string, orderedIds: string[]): Promise<ListingMedia[]>`, `setCover(listingId: string, mediaId: string): Promise<ListingMedia[]>`, `setFloorplan(listingId: string, mediaId: string | null): Promise<ListingMedia[]>`, `remove(listingId: string, mediaId: string): Promise<void>`.
  - `supabase/migrations/0004_listing_media_flags.sql`: add `is_cover bool`, `is_floorplan bool`; partial unique indexes enforcing at most one cover and one floorplan per `listing_id`. No new RLS surface, the table policies from 0003 already cover these columns.
  - `lib/listings/actions.ts`: `createListing` and `updateListing` read the ordered media id list, cover, and floorplan from the form and call `MediaService.reorder` / `setCover` / `setFloorplan`; each action keeps `authService.requireAdmin()` as its first line and the existing `revalidatePath('/')` and `revalidatePath('/listing/${id}')`.
- Seams: existing, action tests through `FakeMediaService` (records `reorderCalls` / `setCoverCalls` / `setFloorplanCalls` / `removeCalls` arrays and returns the resolved set, matching the `Fake*Calls` convention in `tests/listings/actions.test.ts`), asserting the saved order, cover, and floorplan via the recorded arrays and the returned set, not call internals; a `MediaService`-through-`FakeStorageAdapter` test asserts setting a new cover clears the old in the returned set.
- Acceptance: after save, `listForListing` returns media in the submitted order with exactly one `isCover` (first if none chosen) and at most one `isFloorplan`; reopening edit reflects the same; setting a new cover yields a returned set with the previous cover cleared, and tagging a second floorplan yields a set with the first cleared.
- Blocked by: Unit 1, Unit 2

## Unit 5: Publish guard counts uploaded media

- What it builds: the publish predicate is redefined so a listing is publishable when `mediaCount + photoUrls.length >= 1`, and both callsites are rewired. This is the unit that owns US 13 / the "guard counts uploaded images" acceptance, which no other unit delivered.
- Modules + contracts:
  - `lib/listings/schema.ts`: split the photo requirement out of `ListingLiveSchema` so `validateForPublish` no longer hard-requires `photoUrls.min(1)`; introduce `function hasPublishableImage(mediaCount: number, photoUrls: string[]): boolean` returning `mediaCount + photoUrls.length >= 1`. `validateForPublish` keeps validating every non-photo field; the photo requirement is enforced by `hasPublishableImage` at the callsite, since the count now comes from two sources (the table plus legacy URLs).
  - `lib/listings/actions.ts`, `parseListingFormData` (the `intent==='live'` branch): after non-photo validation passes, call `MediaService.listForListing(id)` for the listing being saved (where an id exists) and compute `hasPublishableImage(media.length, photoUrls)`; on false, return a `photoUrls` field error via the existing `humanMessageFor` path. For create (no id yet) the predicate falls back to `photoUrls.length`, since no media row can exist before the listing does.
  - `lib/listings/actions.ts`, `publishListing`: today it re-validates from a pure in-memory candidate built off `current.photoUrls`. Make it async-fetch `MediaService.listForListing(id)` before validating, then gate on `hasPublishableImage(media.length, current.photoUrls)` in addition to `validateForPublish` for the non-photo fields; redirect with the existing `missing-fields` message including `photoUrls` when the combined count is zero.
- Seams: existing, pure-function tests on `hasPublishableImage` (the `schema.test.ts` pattern); plus action tests through `FakeMediaService` (seeded with N media for the listing) and `FakeListingService`, asserting the redirect/field-error outcomes, matching the `tests/listings/actions.test.ts` convention.
- Acceptance: `hasPublishableImage` returns true for `(1, [])`, true for `(0, ['https://x/y.png'])`, false for `(0, [])`; a `publishListing` test with `FakeMediaService` holding one media row and empty `photoUrls` publishes (no missing-fields redirect); a test with one legacy `photoUrls` entry and zero media publishes; a test with zero media and empty `photoUrls` blocks with a `photoUrls` missing-field redirect; the `intent==='live'` save path mirrors all three.
- Blocked by: Unit 1

## Unit 6: One render boundary, gallery and cards render variants and tags

- What it builds: image resolution collapses behind a single `MediaService` boundary so no page branches on `ListingMedia[] | string[]`; the public gallery, the homepage card, the listing detail, and the admin dashboard render the web-optimised main image and thumbnail strip, lead with the cover, and badge the floorplan. A batched lookup avoids a per-card N+1. Frontend, implemented directly, no TDD, except the pure mapper.
- Modules + contracts:
  - extend `lib/media/contract.ts`: `interface GalleryImage { webUrl: string; thumbUrl: string; isFloorplan: boolean }`; `interface MediaService` gains `galleryFor(listing: Listing): Promise<GalleryImage[]>` (single listing, cover first, internally falls back to `listing.photoUrls` when no media rows exist) and `coverFor(listingIds: string[]): Promise<Map<string, GalleryImage | null>>` (one batched call resolving each listing's cover, falling back to its first `photoUrls` entry, used by the homepage and dashboard grids to avoid an N+1). Pages call these; pages never see the `ListingMedia[] | string[]` union.
  - `lib/media/gallery.ts`: pure mapper `toGalleryImages(media: ListingMedia[], photoUrls: string[]): GalleryImage[]` that returns mapped uploaded media when present (cover first, floorplan flagged), else normalises legacy `photoUrls` into `GalleryImage[]` (web and thumb both the legacy URL, `isFloorplan: false`). This is the single fork point, owned in one pure function, tested in isolation; `galleryFor`/`coverFor` are thin callers of it.
  - `components/listing/PhotoGallery.tsx`: prop widened from `photoUrls: string[]` to `media: GalleryImage[]`, ordered cover first, floorplan badged; a plain `<img>` per house rule, never `next/image`.
  - `components/listing/ListingCard.tsx`: today reads `listing.photoUrls[0]` directly (line 10). Switch to a resolved `cover: GalleryImage | null` prop (its `webUrl`), so uploaded-media listings show their real cover instead of a blank card.
  - `components/admin/ListingRow.tsx`: if it surfaces a thumbnail, switch to the resolved cover prop on the same batched path; if it renders no image, no change.
  - `app/listing/[id]/page.tsx` (via `components/listing/ListingDetail.tsx`): resolve `galleryFor(listing)` and pass `GalleryImage[]` to `PhotoGallery`. `app/page.tsx` and `app/admin/(authed)/page.tsx`: resolve covers with one `coverFor(ids)` call and pass each card its resolved cover. Components import no service.
- Seams: the pure mapper `toGalleryImages` gets a pure-function test (an uploaded-media set normalises cover-first with the floorplan flagged; an empty-media-plus-legacy-`photoUrls` set normalises into `GalleryImage[]` through the same return). Components and pages are implemented directly per house rule.
- Acceptance: a listing with uploaded media renders web and thumb variants with the cover first and a badged floorplan; a listing with no media but a non-empty `photoUrls` still renders its external images through the same path, with no blank gallery; the homepage and dashboard render each card's resolved cover (uploaded cover when present, else first legacy URL) via the single batched `coverFor` call, so `ListingCard` never shows a blank cover for an uploaded-media listing and the grid makes one media lookup, not one per card.
- Blocked by: Unit 3, Unit 4

## Unit 7: Upload UI, multi-select, drag-drop, drag-to-reorder, progress

- What it builds: replaces `ImageUrlList` with a real uploader: multi-file select, drag-drop a batch, per-file progress and success or reasoned rejection, drag-to-reorder thumbnails, mark cover, tag floorplan, remove. Frontend, implemented directly, no TDD. The upload action it calls already exists from Unit 1.
- Modules + contracts:
  - `components/admin/ImageUploadList.tsx` (new): props `{ media: ListingMedia[]; onUpload: (files: File[]) => void; onReorder: (orderedIds: string[]) => void; onSetCover: (id: string) => void; onSetFloorplan: (id: string | null) => void; onRemove: (id: string) => void }`; presentation and interaction only, replaces `ImageUrlList` in `ListingForm.tsx`. It calls the existing `uploadListingImage` action (Unit 1) per file and shows each file's typed `UploadResult` as progress to success or a reasoned rejection.
  - remove `components/admin/ImageUrlList.tsx` and the `photoUrls` text-paste input from the form; `parsePhotoUrls` and `serializePhotoUrls` stay only on the legacy read path. This removal is gated through `/destructive-change-gate`: legacy `photoUrls` data is preserved and still renders via Unit 6, no migration drops or rewrites `photo_urls`, only the authoring UI is swapped; confirm before deleting `ImageUrlList.tsx` (verified only `ListingForm.tsx` references it) in case another route references it.
- Seams: existing, no new server action (Unit 1 owns `uploadListingImage`). The React component is implemented directly, no TDD per house rule.
- Acceptance: selecting or dragging multiple files uploads each via the existing action with visible progress and a per-file success or reasoned rejection, valid files in a batch storing even when a sibling is rejected; thumbnails drag to reorder; cover and floorplan can be set; remove drops an image; publishing with zero images is still blocked by the Unit 5 guard counting media plus legacy URLs.
- Blocked by: Unit 2, Unit 4, Unit 5, Unit 6
