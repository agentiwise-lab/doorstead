## Module interaction

One direction only, matching `doorstead/CLAUDE.md`. Buyer routes and server actions (`app/(buyer)/**`, `app/auth/**`, `lib/saved-listings/actions.ts`) import contract types and their typed singletons only, never `@supabase/*`.

- `lib/auth/contract.ts` gains a buyer gate (`requireBuyer()`, `signInWithGoogle()`); `DefaultAuthService` stays the only auth caller of `lib/db/*`. `requireBuyer()` is the buyer analogue of `requireAdmin()`: it returns the session for a signed-in NON-admin and throws otherwise (and fails CLOSED on an admin-check error — see Unit 1), so every buyer surface has one gate. Rejecting admins is a deliberate product choice: admins use admin surfaces, not buyer surfaces.
- `lib/saved-listings/contract.ts` (new) → `DefaultSavedListingService` (`lib/saved-listings/service.ts`) is the only caller of the new `saved_listings` table. It uses the authenticated caller's `createServerClient()` (carries the auth cookie) so RLS keyed on `auth.uid()` scopes every read/write to the signed-in buyer; the buyer never passes their own id, RLS derives it. The shortlist read resolves listing detail in ONE join query inside `DefaultSavedListingService` (`saved_listings` joined to `listings`, under the buyer's authenticated client), reading `listings` through a new authenticated-role SELECT policy scoped to live listings that Unit 1 adds (mirroring `listings_public_read`). No service→service import is needed for the shortlist read: `DefaultSavedListingService` does not depend on the `ListingService` contract.
- `lib/inquiries/contract.ts` gains `listForBuyer(email: string)`; `DefaultInquiryService` reads through `createServerClient()` under a new additive authenticated-SELECT RLS policy that gates on a provider-verified email claim (see Unit 3). The existing anon-write / no-anon-read / admin-read-all policies are untouched, but note the effective inquiries SELECT boundary WIDENS from "admin-only" to "admin OR verified-self-email" because Postgres OR-combines permissive policies; the buyer predicate is a primary PII gate, not a convenience.
- UI components in `components/**` receive data + callbacks as props and import no service, no DB (per `CLAUDE.md`). The save control is a client component fed `{ isSaved, onSave, onUnsave }`; the shortlist and my-inquiries pages are server components that resolve data through the contracts and pass it down.
- Public routes keep using the anon client for the live listings read; only buyer-scoped reads/writes use the cookie-aware server client. Never mixed.

## Data model note (for planning; SQL lives in units, not here)

Today there is no buyer-owned data. This feature adds one table, `saved_listings`, one row per `(user_id, listing_id)` with a UNIQUE constraint on the pair (idempotent save, never a duplicate shortlist entry), `listing_id` referencing `listings(id) on delete cascade` so unlisting removes the shortlist row. RLS: a signed-in user may SELECT / INSERT / DELETE only rows where `user_id = auth.uid()`; there is no admin policy (admin is untouched). The same migration also adds an authenticated-role SELECT policy on `listings` scoped to live listings (see B1 / Unit 1) so a buyer's authenticated client can read the listing detail its shortlist rows point at. Inquiries gain NO `user_id` column: "my inquiries" is matched by the buyer's provider-verified email claim against `inquiries.email` via a new additive SELECT policy, which by construction covers inquiries sent before the account existed and while logged out. Because Postgres OR-combines permissive policies, adding this policy WIDENS the effective inquiries SELECT boundary from "admin-only" to "admin OR verified-self-email" — an intended change, with the buyer predicate as a primary PII gate (see Unit 3).

## Google OAuth provisioning (environment prerequisite, documented in Unit 1)

Buyer auth is Sign in with Google via Supabase's Google OAuth provider. Enabling it is an environment step, not code. It is NOT analogous to provisioning a Storage bucket: a bucket does not mint credentials, but the Google provider does.

### Security precondition: opening public self-signup

Enabling the Google provider makes public self-signup LIVE for the first time. Today the only way to obtain an `authenticated` JWT is admin-provisioned password sign-in; migration `0001` documents the threat explicitly ("the 'authenticated' role is granted to ANY signed-in Supabase user, including a self-signed-up stranger") and the README first-deploy runbook deliberately disables email self-signup and flags it CRITICAL. Turning on Google OAuth means every `to authenticated` RLS policy is now evaluated against an adversary-controllable principal, not just the two admins. Treat provider enablement as a reviewed security decision, not a benign toggle.

Required audit step before cutting issues: enumerate every `to authenticated` policy in the schema (both migrations plus this feature's two new policies) and prove each gates on the `admins` table, on `auth.uid()` ownership, or on a verified-email match — NEVER on role membership alone. Confirm the runbook's email-signup lockdown stays in force and that a Google sign-up cannot mint an admin (the `admins` table is seeded only by the one-time local service-role step; there is no self-service path into it).

### Provisioning checklist (implementer performs in dashboards before the tracer can pass)

Unit 1 must ALSO add this as a first-deploy runbook step in the README (which today covers the email-signup lockdown and admin seed but has no OAuth step), since the human provisioner reads the README, not a migration comment. The runbook step enumerates:

- Enable the Google provider in Supabase → Authentication → Providers → Google.
- Paste the Google Cloud OAuth client id + secret (these live in Supabase project config; no secret is committed, and there is no `.env.example` change).
- Register the exact redirect/callback URLs: the app's `http://localhost:3000/auth/callback` and `https://<prod-domain>/auth/callback`, plus Supabase's own `.../auth/v1/callback` in the Google Cloud console. Add the app callback URLs to Supabase's redirect allowlist.
- Keep email self-signup OFF (do not re-enable it while enabling Google).
- Confirm Google sign-up cannot mint an admin.

State the intent in the plan; do not silently assume the provider is on. Provider provisioning happens BEFORE any unattended `/implement` run and is not part of any unit's automated acceptance (see Unit 1 acceptance split).

## Tracer bullet

A visitor clicks "Sign in with Google", completes Google OAuth, and lands back signed in as a buyer → on ONE live listing they click Save → a `saved_listings` row is written under their `auth.uid()` via RLS → a `/shortlist` page (guarded by the `(buyer)` route-group layout) reads their saved rows via a single join and shows that one listing. Thinnest path through every layer: Google auth session + `saved_listings` table/RLS + the authenticated-role `listings` read policy + a save action gated by `requireBuyer` + a buyer-scoped shortlist join read + a guarded page. No unsave, no saved-state on the control, no my-inquiries, no sign-out yet. The live Google round-trip is a manual post-provisioning smoke check; the unattended loop keys on the automated gate (Fake-backed tests + migration applies locally + local-Postgres RLS assertions + typecheck/build). This is Unit 1.

## Unit 1: Tracer — sign in with Google, save one live listing, see it on the shortlist

- What it builds: a visitor signs in with Google and becomes a buyer session; on a live listing detail page a Save control writes one `saved_listings` row scoped to their `auth.uid()`; a `/shortlist` page lists that saved listing.
- Modules + contracts:
  - `lib/auth/contract.ts`: add `signInWithGoogle(redirectTo: string): Promise<{ url: string }>` (returns the provider authorize URL to redirect the browser to) and `requireBuyer(): Promise<Session>` (returns the session for a signed-in non-admin; throws `Unauthorized` if there is no session, if the user is an admin — admins use the admin surfaces, not buyer surfaces — or if the admin check errors, i.e. fails closed; see the service note below).
  - `lib/auth/service.ts`: `DefaultAuthService` implements both. `signInWithGoogle` calls `client.auth.signInWithOAuth({ provider: 'google', options: { redirectTo } })` and returns its url. `requireBuyer` calls `getSession()` (which already uses `getUser()` to verify the JWT), then checks admin status and throws if admin. It must FAIL CLOSED on an admin-check error: an errored admin check must never be treated as a valid buyer. Today `isAdmin()` swallows a query error as `false` (`service.ts` line 47), which is fail-closed for `requireAdmin` but fail-OPEN for `requireBuyer` (a transient error would misclassify a real admin as a buyer). Resolve this so the buyer gate throws on error — either make `isAdmin` surface errors instead of collapsing them to `false`, or give `requireBuyer` a distinct admin-check path that throws on error. `requireBuyer` rejects admins by design (admins use admin surfaces).
  - `lib/auth/actions.ts`: add `startGoogleSignIn(formData: FormData): Promise<void>` — server action that computes the callback URL and `redirect()`s to the provider url. Add `app/auth/callback/route.ts` (route handler): exchanges the OAuth `code` for a session cookie (`client.auth.exchangeCodeForSession(code)`) then redirects to a validated `next` param or falls back to `/shortlist`. `next` is validated to same-origin RELATIVE paths ONLY: it must start with a single `/` and reject protocol-relative `//` (and anything not starting with `/`); anything else falls back to `/shortlist`. A missing/invalid `code` redirects to the sign-in entry and sets no cookie. The app callback URL must be registered in Supabase's redirect allowlist (see provisioning checklist). This route handler may call `lib/db/server-client.ts` directly for the code exchange, the one server-side auth touchpoint the middleware pattern already establishes.
  - `lib/saved-listings/contract.ts` (new): `interface SavedListingService { save(listingId: string): Promise<void>; listForBuyer(): Promise<Listing[]> }`. `save` is idempotent (upsert on the unique pair; saving twice yields one row). `listForBuyer` returns full `Listing` objects (resolved via one join query) ordered newest-saved-first. No `userId` parameter — RLS derives the owner from `auth.uid()`.
  - `lib/saved-listings/service.ts` (new): `class DefaultSavedListingService implements SavedListingService`, the only caller of the `saved_listings` table, using `createServerClient()`. `listForBuyer` performs a SINGLE join query (`saved_listings` joined to `listings`) under the buyer's authenticated client, relying on both the `user_id = auth.uid()` policy on `saved_listings` and the new authenticated-role live-listings SELECT policy on `listings` (added by this unit's migration), and maps the result to the existing `Listing` shape. It does NOT import the `ListingService` contract and performs no cross-service call.
  - `lib/saved-listings/actions.ts` (new): `'use server'` — `saveListing(formData: FormData): Promise<void>`, first line `await authService.requireBuyer()`, then `savedListingService.save(listingId)`, then `revalidatePath('/shortlist')` and `revalidatePath('/listing/${id}')`.
  - `components/listing/SaveControl.tsx` (new, `'use client'`): props `{ listingId: string; isSaved: boolean; onSave: () => void; onUnsave?: () => void }`. In Unit 1 only Save is wired (a form posting `saveListing`); `isSaved` and `onUnsave` are part of the contract but exercised in Unit 2. No service import.
  - `components/listing/ListingDetail.tsx`: accept an optional `saveControl?: ReactNode` slot and render it; the page decides whether to pass one (only when a buyer session exists).
  - `app/listing/[id]/page.tsx`: if a buyer session exists (`getSession()` + not admin), render the `SaveControl` into the detail. Public read still uses the anon client for the listing itself.
  - `app/(buyer)/layout.tsx` (new): the shared buyer route-group guard — one `requireBuyer()` gate for the whole `(buyer)` group, mirroring `app/admin/(authed)/layout.tsx`. Wrap `requireBuyer()` in try/catch and `redirect()` to the sign-in entry on throw, so `/shortlist` (and later `/my-inquiries`) are born guarded: an unauthenticated or admin request yields no data and is directed to sign in. RLS remains defense-in-depth (a null uid reads zero rows), never the only gate.
  - `app/(buyer)/shortlist/page.tsx` (new, `force-dynamic`): the route-group layout already guards it; the page reads the signed-in buyer's data via `savedListingService.listForBuyer()` and renders the saved `Listing`s reusing `ListingCard`.
  - A minimal sign-in entry point: a "Sign in with Google" control (form posting `startGoogleSignIn`) surfaced on the public header or the listing detail so the tracer is reachable end to end. Full auth entry-point polish (header sign-in/out controls) is Unit 4; Unit 1 needs just enough to demo the path plus the layout guard above.
  - migration (the **saved_listings migration** — pick the next unused numeric prefix at implement time by running `ls supabase/migrations/`; do NOT hardcode `0003`, because the parallel `listing-image-uploads` branch also introduces a `0003_*` migration from the same lineage. Merge rule: whichever feature merges second renumbers its migration before `supabase db push`). Contents: create `saved_listings (id uuid pk default gen_random_uuid(), user_id uuid not null references auth.users(id) on delete cascade, listing_id uuid not null references listings(id) on delete cascade, created_at timestamptz not null default now(), unique (user_id, listing_id))`; `alter table ... enable row level security`; three policies `to authenticated` for SELECT / INSERT / DELETE, each `using (user_id = auth.uid())` / `with check (user_id = auth.uid())`. No anon policy, no admin policy. ALSO in this migration, ADD an authenticated-role SELECT policy on `listings` scoped to live listings, mirroring the existing `listings_public_read`: `for select to authenticated using (status = 'live' and deleted_at is null)`. Without it a buyer's authenticated client matches no `listings` SELECT policy (the existing `listings_admin_all` requires membership in `admins`) and the shortlist join returns zero rows. Include the Google-provider provisioning + public-self-signup security note (above) as a comment header / doc note in this unit so an unattended run knows the dashboard step is a prerequisite. The unattended loop runs `supabase db push` only against a LOCAL stack; a human runs/re-verifies the remote push during provisioning (additive residue on revert is harmless).
- Seams: a `Fake*` seam at the `SavedListingService` contract (`tests/saved-listings/fakes.ts`, mirroring `tests/inquiries/fakes.ts`). The action test drives `saveListing` against `FakeSavedListingService` and a fake auth that returns a buyer session, asserting a save is recorded (this proves delegation/behaviour, not the DB-enforced guarantees). Auth-gate behaviour (`requireBuyer` throwing for admin / no session, and failing closed on admin-check error) is tested through the `AuthService` contract with a fake session source. The DB-enforced guarantees (cross-buyer isolation, idempotency) are proven by real-Postgres assertions in the acceptance, not by the fakes.
- Acceptance:
  - **(a) Automated (the unattended loop's pass/fail gate):**
    - Fake-backed contract + action tests pass (`saveListing` gated by `requireBuyer` + delegation; `requireBuyer` throw-paths including fail-closed-on-error).
    - The saved_listings migration applies against a local `supabase start` stack; typecheck and `next build` pass.
    - Local-Postgres RLS assertions against the local stack: buyer A cannot SELECT / INSERT / DELETE buyer B's `saved_listings` rows; inserting the same `(user_id, listing_id)` twice yields exactly one row; a buyer's authenticated client can SELECT a live listing (confirming the new authenticated-role `listings` policy).
  - **(b) Manual post-provisioning smoke (NOT run by the loop):** after a human enables the Google provider (per the provisioning checklist), one live Google sign-in → save one live listing → see it on `/shortlist`. Provider provisioning happens before the unattended run and is not part of any unit's automated acceptance.
- Blocked by: none, parallel-safe

## Unit 3: My inquiries view (email-matched, additive RLS)

- Modules + contracts:
  - migration (the **inquiries buyer-read migration** — pick the next unused numeric prefix at implement time via `ls supabase/migrations/`; do NOT hardcode `0004`. Same merge rule: whichever feature merges second renumbers before `supabase db push`). ADD one policy `inquiries_buyer_read` that gates on a provider-VERIFIED email claim, for example `for select to authenticated using ((auth.jwt() ->> 'email_verified')::boolean is true and email = auth.jwt() ->> 'email')`. The exact claim path (`email_verified`, `email`, or a nested `user_metadata`/`app_metadata` location) MUST be confirmed against a real decoded Google JWT for THIS project before implementation, not taken from docs. ## Requirement coverage map

Every PRD requirement maps to a unit:

- Accounts/sign-in: Google sign-in without password → U1 (control + flow) polished in U4; verified email is the identity everywhere → U1 (`requireBuyer`/session), U3 (verified-email match); sign-out → U4; buyer gains no admin capability → U4 (restated/verified), enforced by `requireBuyer` U1 (fail-closed, rejects admins) + existing `requireAdmin`.
- Shortlist save/unsave: save from overview → U2; save from detail → U1; unsave from either → U2; idempotent single entry → U1 (verified by the local-Postgres assertion that a duplicate `(user_id, listing_id)` insert yields exactly one row — the unique constraint enforced in the DB, not the fake); control reflects state on load → U2; unauthenticated save/unsave creates/modifies nothing → U1 (`requireBuyer` gate + the `(buyer)` layout guard + RLS, with cross-buyer isolation verified by the local-Postgres assertion).
- Shortlist view: page lists exactly saved listings → U1 (single join read under the buyer client + the new authenticated-role `listings` policy); empty state → U1 (empty render) reinforced by acceptance; navigate to detail from shortlist → U1 (reuses `ListingCard` link); unauthenticated shortlist yields no data + directs to sign in → U1 (`(buyer)` layout guard).
- My inquiries: exact verified-email match, no others → U3 (verified by the local-Postgres assertion that a buyer JWT reads only its own-verified-email inquiries and zero others); includes pre-account and logged-out inquiries → U3 (email-claim RLS); empty state → U3; unauthenticated yields no data + directs to sign in → U1 (`(buyer)` layout guard covers the whole group).
- Authorization boundaries: no admin screen for buyer → U4 (verified against existing guard); one buyer cannot read another's shortlist → U1 (`user_id = auth.uid()` RLS, verified by the local-Postgres cross-buyer-isolation assertion); cannot read inquiries for another email → U3 (verified-email-claim RLS, verified by the local-Postgres email-scoped assertion).
- Anonymous funnel unchanged: anon can still inquire, no sign-in prompt → U3 (leaves `submitInquiry`/anon-insert untouched; asserted, plus a local-Postgres assertion that anon still cannot SELECT inquiries); inquiry PII boundary WIDENS by design from "admin-only" to "admin OR verified-self-email" → U3 (additive policy OR-combined with `inquiries_admin_read`; the buyer predicate is a primary PII gate verified in the DB, and `inquiries_admin_read` / anon-no-SELECT are untouched).
- Known edge case (accepted, inquiry email column is unverified free text) → U3. Because the buyer policy gates on the provider-VERIFIED JWT email claim, an authenticated buyer can only ever read rows whose `inquiries.email` equals their own verified address; the residual accepted edge (someone else typing your address into an inquiry) surfaces only rows carrying the buyer's own address, no third-party contact details. Documented as accepted; the security-critical part (verified-claim gating) is pinned in U3, not deferred.

No requirement is unmapped. The feature is additive at the schema level (new table, two new RLS policies, new pages, new controls, extended contracts) with no dropped/weakened policy, so the destructive-change gate is not triggered. Note, however, that the effective inquiries SELECT boundary intentionally widens (admin → admin OR verified-self-email) and public self-signup goes live when the Google provider is enabled; both are reviewed decisions recorded above, not silent side effects.
