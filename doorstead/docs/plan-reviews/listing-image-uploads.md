# Plan Review: listing-image-uploads

Altitude: unit breakdown with contracts and sequencing (7 tracer-first units, one feature/branch). Reviewed at that altitude; implementation-level detail the plan deliberately left open was not flagged.

## Findings

### Public render path cannot mint a signed URL for a private-bucket object — the tracer bullet's acceptance is unreachable as specified
- Severity: blocker
- Confidence: high
- Evidence: The tracer (Unit 1) requires the **public** listing detail page to render `<img src={signedUrl}>` for an object in a **private** bucket, and the migration line explicitly says "no anon direct object read; public reads go through signed links minted server-side." Verified against Supabase docs: for a private bucket, "the only ways to download assets ... Create a signed URL with the `createSignedUrl` method"; and access to `storage.objects` (including the SELECT that `createSignedUrl` performs) "is subject to access control via RLS policies" — "Service keys entirely bypass RLS policies" (Storage Access Control, Storage Buckets fundamentals). Verified against the codebase: both DB clients (`lib/db/anon-client.ts`, `lib/db/server-client.ts`) are built from `NEXT_PUBLIC_SUPABASE_ANON_KEY`; there is no service-role client. `.env.example` states these "are the ONLY env vars the running app needs" and that `SUPABASE_SERVICE_ROLE_KEY` "is never added to .env, never set in Vercel, and never deployed anywhere." The public page (`app/listing/[id]/page.tsx`, `export const dynamic = 'force-dynamic'`) reads via the anon-keyed singleton. So at render time the code holds only the anon role, the migration denies that role SELECT on `storage.objects`, and there is no service key to bypass RLS. `createSignedUrl` will therefore fail (or return an unusable/empty result) on the public path, and the acceptance criterion "loading the listing detail page renders an `<img>` whose src is a signed link that resolves to those bytes" cannot be met. This is not implementation detail: it is a contradiction between the security model in the migration bullet and the runtime credentials the app is allowed to have.
- Recommendation: Decide the signing credential explicitly in Unit 1 and make it a named contract obligation, not a migration footnote. Two coherent options: (a) grant the **anon** role a narrow SELECT policy on `storage.objects` scoped to this one bucket so the anon client can mint signed URLs (objects still are not publicly enumerable because the raw object path stays private and only time-limited signed links are handed out — this satisfies user story 17 / the "not publicly enumerable" criterion); or (b) introduce a service-role storage client in `lib/db/storage.ts` and the env/deploy change that entails — but that directly contradicts the app's stated "anon-key-only in the running app" invariant and README first-deploy runbook, so it is the more expensive path and must be called out. Pick (a) unless there is a reason not to, and write the chosen SELECT/`createSignedUrl` credential into Unit 1's module-and-contract list so no later unit re-litigates it.

### `MediaService` singleton has no defined DB-client wiring, unlike every existing service
- Severity: major
- Confidence: high
- Evidence: Every existing service resolves its client explicitly: `authService` uses `createServerClient()` internally; `DefaultListingService` is constructed with `anonClient` for public reads but calls `createServerClient()` inside every admin write/read (`listAll`, `getById`, `create`, `update`, ...). The plan says `DefaultMediaService` is "the only caller of `lib/db/storage.ts`" and that `lib/db/storage.ts` "wraps Supabase Storage", but never states which client `storage.ts` uses for each operation. This matters because the two operations have opposite trust contexts: `uploadObject` runs from an admin server action (needs the cookie-aware `createServerClient()` so RLS sees the authenticated admin and the INSERT policy passes) while `createSignedUrl` runs on the public anon path (Finding 1). A single injected client cannot serve both, and the plan does not name the split. Left undecided, the implementer will guess and likely wire one client for both, reproducing Finding 1 or breaking admin upload.
- Recommendation: In Unit 1, specify per-method client selection in `lib/db/storage.ts` the same way `DefaultListingService` does: `uploadObject` → `createServerClient()` (admin/authenticated, INSERT policy); `createSignedUrl` → the client that holds the signing grant decided in Finding 1. State it in the contract note so the seam is unambiguous before implementation.

### `getImagesForRender` is placed on `ListingService` but its impl must call `MediaService` — a service→service dependency the module-direction rule constrains
- Severity: major
- Confidence: medium
- Evidence: Unit 1 adds `getImagesForRender(listingId): Promise<RenderImage[]>` to `lib/listings/contract.ts`, and Units 3/5/6 grow it to merge stored media (signed web/thumb urls, cover/floorplan ordering) with legacy `photo_urls`. Producing that list requires reading `listing_media` and minting signed URLs — i.e. `DefaultListingService` must consume `MediaService`. `doorstead/CLAUDE.md` mandates `routes → service contract`, `service impl → lib/db/*`, and one-directional module flow; `DefaultListingService` today imports only `lib/db` and its own contract. Making it depend on `MediaService` is allowed under the global "services → service contracts" rule only if it imports the **contract**, not the impl, and only if the two services do not become mutually dependent. The plan's "Module interaction" section routes the admin action to both `MediaService` and `ListingService` as siblings, which reads as if they are peers, yet Unit 6 quietly makes `ListingService` the caller of `MediaService`. The ownership of the resolver (listings service vs. a thin route-level composition that calls both service contracts) is never stated, and getting it wrong produces exactly the "route → service → another service impl" tangle the conventions forbid.
- Recommendation: Name the owner of `getImagesForRender` explicitly and keep it contract-only. Cleanest: put the resolver behind `ListingService`'s contract but have its impl depend on the `MediaService` **contract** (injected), so tests still drive it through `FakeMediaService`; OR make image resolution a route-level composition where the page calls `mediaService.listForListing` + reads legacy urls and merges. Either is fine, but choose in Unit 1 (the unit that first introduces `getImagesForRender`), because Units 3/5/6 all extend this same method and a late change ripples through four units.

## Priority-Ranked Actions
1. Resolve the signing-credential contradiction in Unit 1: grant anon a bucket-scoped SELECT on `storage.objects` so the public page can mint signed URLs with the anon key (preferred), and write the credential into the Unit 1 contract. (Finding 1)
2. Specify per-method client selection in `lib/db/storage.ts` (`uploadObject` → server-client; `createSignedUrl` → the Finding-1 grant holder). (Finding 2)
3. Name the owner of `getImagesForRender` and make its impl depend on the `MediaService` contract, not impl, before Units 3/5/6 build on it. (Finding 3)

## Verdict
revise: the tracer bullet's public-render acceptance is unreachable under the app's anon-key-only runtime until the signed-URL credential and per-method storage-client wiring are pinned down in Unit 1; the design and sequencing are otherwise sound (bucket-in-migration is valid, legacy passthrough and unit ordering hold).
